# 实施步骤与阶段

本文档给出平台落地的分阶段步骤，便于按步骤实现。

---

## 阶段 1：定「任务合同」（约 1–2 天）

**目标**：把「一次数学研究任务」和「一次运行」抽象成可存储、可重放的对象。

- **统一任务对象**  
  - `task_id`、目标问题描述、约束条件、预算（时间/API 次数）、停止条件（分数阈值、轮数上限等）。
- **统一运行对象**  
  - `run_id`、所属 `task_id`、阶段状态（draft / evolve / formalize / verify / report）、每步的输入/输出引用（指针到产物）。
- **统一产物对象**  
  - 代码文件、运行日志、数学形式文本、Lean 文件、评估报告等，每个带类型、路径、时间戳。

**产出**：文档化的数据模型（可先用于 SQLite schema），以及平台对「任务 / 运行 / 产物」的 ID 与命名约定。

---

## 阶段 2：做三个适配器（约 3–5 天）

**目标**：用薄封装把 OpenCode、OpenEvolve、Numina 变成平台可调用的组件，不改变其内部实现。

- **OpenCodeAdapter**  
  - 封装：project 初始化、session 创建、发 message、取 message 列表、文件读写/状态。  
  - 输入输出与 OpenCode 的 `specs/project.md` 及 HTTP API 对齐；可选用官方 SDK（如 JS）或直接 HTTP。
- **OpenEvolveAdapter**  
  - 封装：启动一次进化（给定初始程序路径、config、迭代数）、轮询/等待结束、收集 `best_program.py`、checkpoints、logs。  
  - 输出：best 代码路径、评估指标、是否超时/失败。
- **NuminaAdapter**  
  - 封装：调用现有 `translate_LLM.py` 流程（code → math → R1–R7 → Lean），支持 `TRANSLATE_BACKEND=api|numina_agent`。  
  - 输入：代码路径或代码内容；输出：math_form、check 结果、lean 代码、使用的 backend。

**产出**：三个适配器模块，以及简单的单测/脚本验证「各适配器在本地可单独跑通」。

---

## 阶段 3：最小编排流（MVP，约 3–5 天）

**目标**：实现「单任务、串行」的完整流水线，并保证每步状态与产物落盘。

- **流水线步骤**  
  1. 问题 → 草案代码（通过 OpenCodeAdapter 或手填）。  
  2. 草案 → 进化（OpenEvolveAdapter），得到 best 程序。  
  3. best 程序 → 数学形式 + R1–R7 检查（NuminaAdapter）。  
  4. 通过后 → 数学形式 → Lean（NuminaAdapter，agent 优先，失败回退 API）。  
  5. 汇总：报告 + 证据链（代码、math、Lean、评估、诊断）。
- **状态与产物**  
  - 每一步完成后更新 run 的当前阶段，并把产物路径/内容写入 artifacts 表与约定目录；某步失败也写入失败原因与日志，便于重试与审计。
- **断点续跑**  
  - 支持「从某阶段重新开始」（例如从 formalize 重跑），避免重复进化。

**产出**：一个可执行的 `run_pipeline(task_id)`（或等价的单任务入口），加一份「MVP 使用说明」（如何创建任务、如何跑一次、产物在哪看）。

---

## 阶段 4：治理能力（持续迭代）

**目标**：让平台可观测、可控制、可扩展。

- **可观测**  
  - 任务/运行时间线、各阶段耗时、失败率、成本统计（API 调用次数、token 估算）。
- **质量闸门**  
  - 可配置：evaluator 分数阈值、R1–R7 是否必须通过、Lean 错误数上限；不满足时自动标记失败或重试。
- **并发与调度**  
  - 任务队列、优先级、单机/多机并发上限、预算上限（按任务或全局），防止资源耗尽。

**产出**：配置项文档、监控/统计接口或面板、简单调度器（可先内存队列 + 单进程）。

---

## 阶段依赖关系

```
阶段 1（任务合同） ──► 阶段 2（三个适配器） ──► 阶段 3（最小编排流） ──► 阶段 4（治理）
       │                        │                         │
       └────────────────────────┴─────────────────────────┘
                    均依赖统一的任务/运行/产物模型
```

建议：阶段 1 与 2 可部分并行（先定好任务/运行/产物字段，再实现适配器）；阶段 3 必须在 2 完成后；阶段 4 可在 3 跑通后逐步加。
