# 基于 OpenCode 的数学研究多智能体平台 — 项目难点分析

## 概述

大项目目标：以 **OpenCode** 的接口/服务为底座，构建**数学研究场景下的多智能体协同平台**，串联演化 agent（OpenEvolve）、形式化/Lean agent（Numina）、代码生成 agent（OpenCode）等。

本文档归纳实施过程中的主要难点（按优先级）。

---

## 1. 编排一致性（最大难点）

- **OpenCode**：会话式 agent，通过 project/session/message 驱动，状态在服务端。
- **OpenEvolve**：批处理进化流程，本地进程、checkpoint、best 程序路径。
- **Numina**：形式化校验与翻译流水线（code→math→R1-R7→Lean），有 API/agent 双后端。

三者**时序与状态模型不一致**：谁先谁后、输入输出如何传递、失败如何回滚，若没有统一编排层，容易出现「某一步成功但全局不可复现」或状态错乱。

---

## 2. 可复现与可追溯

研究场景必须能回答「这个结论是怎么来的」。需要把：

- 每次调用的 prompt、模型、代码版本
- 评估结果（evaluator 分数、checkpoint）
- 数学形式与 R1–R7 校验结果
- Lean 诊断与最终证明/反例

**全量落盘并可回放**，否则无法做实验对比与审稿。

---

## 3. 长任务可靠性

进化和 Lean 修复都是长流程（可能数十分钟到数小时），需要：

- **任务状态机**：draft → evolve → formalize → lean → report，每步可查、可重试。
- **断点续跑**：某阶段失败后能从该阶段重试，而不是从头再来。
- **超时与重试**：对 LLM/API 调用设超时和重试策略，避免单次抖动导致整条链路失败。

---

## 4. 质量闸门定义

何时算「通过」需要在平台层明确定义：

- 仅 evaluator 分数？
- 必须通过 R1–R7？
- Lean 诊断零错误？还是允许 warning？
- 多目标时如何加权（性能 vs 形式化严谨）？

若不在设计阶段定好，平台行为会不稳定，难以做自动化决策。

---

## 5. 资源与成本控制

多 agent 并行会迅速放大：

- **API 成本**：OpenCode 会话、OpenEvolve 的 LLM 调用、Numina 的翻译/校验。
- **本地算力**：进化评估、Lean 编译。

需要**预算上限、并发限制、配额隔离**（按任务/用户），否则容易超支或拖垮本机。

---

## 6. 接口边界

要明确「平台对外暴露什么」：

- 是**任务/运行/产物**等高层抽象，还是把 OpenCode/OpenEvolve/Numina 的原始接口也暴露？
- 哪些是**内部适配层**（仅平台代码调用），哪些是**稳定 API**（给前端、脚本、其他服务用）？

边界不清会导致 API 膨胀、难以演进。

---

## 小结

| 难点           | 影响           | 应对方向                         |
|----------------|----------------|----------------------------------|
| 编排一致性     | 状态错乱、难复现 | 统一任务/运行/阶段模型与编排引擎 |
| 可复现与可追溯 | 无法审稿与对比   | 全量落盘、时间线、审计接口       |
| 长任务可靠性   | 中途失败全废     | 状态机、断点续跑、超时重试       |
| 质量闸门       | 通过标准不一致   | 显式配置：阈值、多目标权重       |
| 资源与成本     | 超支、拖垮机器   | 预算、并发上限、配额             |
| 接口边界       | API 失控         | 平台只暴露任务/运行/产物/审计     |
