# 存储与目录设计

本文档约定平台**数据与产物的存储位置**及**元数据表结构**，便于实现可复现与审计。

---

## 一、目录布局（建议在仓库根下新增 `platform/`）

在项目根目录（与 `opencode`、`numina-lean-agent` 同级；OpenEvolve 已置于 `opencode/openevolve/`）新增 `platform/`，并采用如下结构：

```
platform/
├── data/
│   ├── db.sqlite              # 任务、运行、状态、产物索引等元数据（可后续迁到其他 DB）
│   └── artifacts/             # 所有大文件类产物，按 task/run 组织
│       └── {task_id}/
│           └── {run_id}/
│               ├── draft/     # 初始草案代码、prompt 等
│               ├── evolve/    # OpenEvolve 的 best_program、checkpoints 索引、logs
│               ├── formalize/ # math_form 文本、R1–R7 检查结果
│               ├── lean/      # .lean 文件、诊断日志
│               └── report/    # 最终报告（json / md）
├── config/                    # 流程策略、阈值、预算、provider 配置
├── prompts/                   # 平台层 prompt 模板（版本化）
└── services/                  # 编排逻辑与 OpenCode / OpenEvolve / Numina 适配器
```

说明：

- **`data/db.sqlite`**：存放任务、运行、阶段、产物索引、事件、指标等，不放大文件内容。
- **`data/artifacts/{task_id}/{run_id}/`**：同一 run 下各阶段产物分目录存放，便于按阶段重放与清理。
- **`config/`**：存放质量闸门、预算、模型选择等，便于不同实验配置切换。
- **`prompts/`**：平台层统一使用的 prompt 模板，与各子项目内部 prompt 分离，便于版本管理与 A/B 测试。

## 二、元数据表结构（最少 6 张表）

以下为建议的 SQLite schema，满足「任务 / 运行 / 阶段 / 产物 / 事件 / 指标」的闭环。

### 1. tasks

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 任务 ID（如 UUID） |
| created_at | TEXT | 创建时间（ISO8601） |
| updated_at | TEXT | 更新时间 |
| goal | TEXT | 目标问题描述 |
| constraints | TEXT | 约束（JSON） |
| budget | TEXT | 预算（JSON：时间、API 次数等） |
| stop_conditions | TEXT | 停止条件（JSON） |
| status | TEXT | 枚举：active / paused / archived |

### 2. runs

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 运行 ID |
| task_id | TEXT FK | 所属任务 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |
| current_stage | TEXT | 当前阶段：draft / evolve / formalize / lean / report |
| status | TEXT | 枚举：pending / running / succeeded / failed / cancelled |
| started_at | TEXT | 开始时间（可为空） |
| finished_at | TEXT | 结束时间（可为空） |
| resume_from_stage | TEXT | 若为续跑，从该阶段开始（可为空） |
| error_message | TEXT | 最近一次失败原因（可为空） |

### 3. stages

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 阶段记录 ID |
| run_id | TEXT FK | 所属 run |
| name | TEXT | 阶段名：draft / evolve / formalize / lean / report |
| status | TEXT | 枚举：pending / running / succeeded / failed |
| started_at | TEXT | 开始时间 |
| finished_at | TEXT | 结束时间 |
| input_refs | TEXT | 输入产物 ID 或路径（JSON 数组） |
| output_refs | TEXT | 输出产物 ID 或路径（JSON 数组） |
| error_message | TEXT | 失败时错误信息 |

### 4. artifacts

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | 产物 ID |
| run_id | TEXT FK | 所属 run |
| stage_name | TEXT | 所属阶段 |
| kind | TEXT | 类型：code / log / math_form / lean / report / config 等 |
| path | TEXT | 相对 `artifacts/{task_id}/{run_id}/` 的路径或绝对路径 |
| created_at | TEXT | 创建时间 |
| meta | TEXT | 可选 JSON（如 mime、大小、checksum） |

### 5. events

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增 |
| run_id | TEXT FK | 所属 run |
| at | TEXT | 发生时间（ISO8601） |
| type | TEXT | 事件类型：stage_started / stage_finished / adapter_call / error 等 |
| payload | TEXT | JSON 详情（如 adapter 名、耗时、状态码） |

### 6. metrics

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 自增 |
| run_id | TEXT FK | 所属 run |
| stage_name | TEXT | 阶段（可为空表示 run 级） |
| name | TEXT | 指标名：score / api_calls / token_count / duration_sec 等 |
| value | REAL 或 TEXT | 数值或 JSON 字符串 |
| recorded_at | TEXT | 记录时间 |

---

## 三、与编排逻辑的配合

- **创建 run**：插入 `runs`，并可为各阶段预插 `stages`（status=pending）。
- **每阶段开始/结束**：更新对应 `stages` 行，并写入 `events`；产物路径写入 `artifacts`。
- **断点续跑**：根据 `run.current_stage` 与 `run.resume_from_stage` 只执行未完成阶段，并从 `artifacts` 取上一阶段输出作为本阶段输入。
- **审计与复现**：通过 `events` + `artifacts` + `metrics` 还原时间线与所有产物引用，必要时再按 path 读取文件内容。

此设计保证「所有文档放进 doc」后，平台实现仍有一致的存储与检索约定，便于按步骤实现和后续扩展。
